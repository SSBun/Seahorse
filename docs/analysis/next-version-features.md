# Seahorse 下一版本功能建议

> 建议版本：1.10.0（小版本假设）
>
> 建议主题：**找得到，删得回**
>
> 生成日期：2026-07-15

## 决策摘要

下一版本建议只交付两个面向用户的核心能力：

1. **智能集合（保存筛选）**：把内容类型、分类、标签、收藏、日期和关键词组合成可复用的动态视图。
2. **回收站与恢复**：让来自 UI、批量诊断和 MCP 的条目删除先进入回收站，内部图片只在永久删除时清理。

这两个功能共同补齐 Seahorse 当前最关键的产品闭环：用户已经能高效采集资料，但资料增长后只能反复手工筛选，误删又不可恢复。智能集合提升“找回和复用”，回收站提供扩大自动化与批量操作所需的安全底座。

下一版本不建议同时加入 Share Extension、OCR、网页全文归档、真正的跨设备同步、协作或更多 MCP 写工具。它们都有价值，但会引入新 target、App Group、索引迁移、存储增长或冲突解决，不适合和本次数据模型变更一起进入一个小版本。

## 前提与证据边界

本报告基于当前 1.9.0 源码、提交历史、构建/架构/数据流/API/数据模型报告，以及同类产品的官方功能说明。当前 GitHub 仓库没有公开 issue，项目也没有产品遥测或用户访谈材料，因此：

- “用户痛点”是根据现有流程和产品定位推断，不等同于已完成用户验证。
- 市场对照只用于确认成熟工作模式，不代表照搬竞品路线。
- 假设下一版本仍是本地优先、单用户、macOS 为主的小版本；若目标改为正式发布 iOS 或云同步，应重新排序。

## 当前产品闭环

| 环节 | 已有能力 | 已确认缺口 |
|---|---|---|
| 采集 | 窗口粘贴/拖放、双击复制、手工新增、浏览器书签导入、MCP 创建 | 无系统 Share Extension；占位 OGP 与自动 AI 可并发覆盖，AI 忙时漏掉新增事件 |
| 整理 | 分类、标签、收藏、AI 摘要/分类/标签、批量解析 | 没有保存筛选或动态集合；默认分类与部分规则依赖英文名称 |
| 检索 | title/URL/notes/content/path/tag 搜索，类型/分类/标签/收藏与排序 | 筛选是临时状态；图片不可按画面文字搜索；普通搜索不含 metadata description/siteName/category |
| 再使用 | 网页/图片/Markdown 详情、复制、打开、Agent 搜索、移动端静态页 | 没有“重新发现/复习”机制；Agent 只覆盖 Bookmark |
| 安全 | 原子写单个 JSON、手工备份/恢复、链接诊断 | 删除立即永久生效；跨 JSON/Images 无事务；解码失败缺少用户恢复入口 |
| 跨设备 | 同一 target 有 iOS 浏览与导入；可生成移动端 HTML | 没有实时双向同步、冲突解决或 iOS 发布流水线 |

最近 1.7.0—1.9.0 的主要增量集中在 MCP 搜索、读取、更新、删除与 helper 可靠性。下一版本若继续只扩张 agent 接口，会让自动化能力领先于普通用户的资料整理与安全体验。

## 市场校准

同类工具的官方资料显示，成熟产品通常围绕四类任务形成差异化：

| 产品 | 已验证能力 | 对 Seahorse 的启示 |
|---|---|---|
| Raindrop.io | 全文搜索、网页存档、提醒、重复/失效链接检查和自动备份；筛选支持类型、日期、无标签、提醒等组合。来源：[Pro 功能](https://help.raindrop.io/premium-features)、[筛选](https://help.raindrop.io/filters) | Seahorse 已有失效链接检查，但“长期可检索”和“资料库健康”仍应继续补齐。 |
| Anybox | Smart List 按类型、标签和日期自动组织；支持 CloudKit、剪贴板检测、URL Scheme。来源：[Getting Started](https://anybox.app/getting-started) | 对原生 Apple 平台工具，动态列表是资料量增长后的基础能力；自动化入口应建立在可靠本地模型之上。 |
| Eagle | Smart Folder、丰富条件筛选、批量动作，以及重复图片扫描和 metadata 合并。来源：[Why Eagle](https://eagle.cool/blog/post/why-eagle)、[重复图片扫描](https://en.eagle.cool/blog/post/no-more-duplicate-images-eagles-scanning-for-identical-files-frees-up-your-hard-drive) | 面向素材用户，动态整理与安全批量处理比继续增加单点编辑功能更有价值。 |
| mymind | AI 图片标签、图片文字识别、Smart Spaces，以及让旧内容重新出现的 Serendipity。来源：[功能与定价](https://access.mymind.com/pricing)、[移动端与 OCR](https://mymind.com/mobile)、[Serendipity](https://mymind.com/clear-your-mind) | OCR 与重新发现值得进入后续路线，但应先建立可复用筛选、稳定索引和查看状态。 |

这些产品共同说明“采集速度”只是入口；当资料库增长后，用户价值来自自动组织、深度检索、重新发现和安全维护。Seahorse 当前最容易以既有架构补齐的是智能集合与回收站。

## 下一版本详细范围

### 功能一：智能集合（保存筛选）

#### 用户问题

当前筛选只存在于页面状态。用户每次想查看“最近一周未整理的图片”“收藏的设计类资料”或“带任一项目标签的笔记”，都要重新选择范围、类型、标签和排序。资料越多，分类和单标签导航越难表达跨维度任务。

#### 最小可用范围

- Sidebar 新增“智能集合”区，支持创建、重命名、编辑、删除和排序。
- 支持以下条件：
  - 内容类型：全部 / Bookmark / Image / Text。
  - 分类：一个分类或全部。
  - 标签：一个或多个，明确选择“任一”或“全部”。
  - 收藏状态。
  - 添加日期：今天、最近 7 天、最近 30 天、自定义起止日期。
  - 文本关键词。
- 支持保存排序方式，并显示实时结果数量。
- 提供两个无需用户配置的系统视图：
  - “最近添加”：最近 7 天。
  - “待整理”：`None` 分类或无标签的条目。
- 智能集合随完整导出/导入保存；macOS 和 iOS 使用同一查询模型，iOS 至少可读取并打开结果。

#### 最小数据与代码改动

- 新增 `SmartCollection`：`id`、`name`、`criteria`、`sortOrder`、`createdAt`。
- 扩展 `CollectionSearch.Criteria`：日期范围、tag match mode、无标签/待整理条件；继续保持纯函数和三端共享。
- 新增 `smart-collections.json` 或等价的明确持久化集合，并纳入 schema version、导入导出和 `DataStorage`。
- Sidebar 与现有 iOS Filter 复用同一条件模型；不要创建第二套查询引擎。

#### 边界情况

- 规则引用的分类/标签被删除：保留智能集合并显示“条件已失效”，要求用户编辑；不能把缺失条件静默解释为“全部”。
- 分类/标签重命名：规则按 UUID 继续有效，显示新名称。
- 多标签语义：UI 和测试必须区分 ANY/ALL，避免沿用当前不透明的 OR 行为。
- 特殊范围：`All Bookmarks`、`Favorites`、`None` 不应靠可重命名名称持久化为业务规则；使用明确 enum/条件。
- 导入冲突：同 ID 更新或跳过，同名不同 ID 提示而非静默创建重复视图。
- 空结果：提供编辑条件入口，不自动删除智能集合。

#### 非目标

- 不支持任意嵌套的 AND/OR 条件树。
- 不执行“匹配后自动改标签/移动分类”等动作。
- 不让 AI 自动生成规则。
- 不加入共享、云同步或团队权限。

#### 验收门槛

- 同一条件在 macOS、iOS 和核心搜索测试中结果一致。
- 规则保存、重启、导出、导入后结果不变。
- 删除被引用的分类/标签不会崩溃，也不会扩大匹配范围。
- 1,000 条本地数据下，条件变化到首批结果显示保持在可感知即时范围；沿用现有搜索 benchmark 建立回归阈值。
- 至少用 5 个真实任务验证能在 30 秒内创建目标视图；若没有遥测，不需要为了这一版新增追踪 SDK。

### 功能二：回收站与恢复

#### 用户问题

当前卡片、列表、详情、失效链接批量清理和 MCP `delete_item` 都直接永久删除。ImageItem 的内部文件同时被移除，操作无法撤销。随着 batch 和 agent 能力扩大，单次误操作的损失也随之扩大。

#### 最小可用范围

- 所有条目删除先写入回收站，不立即清理内部图片。
- Sidebar 新增“回收站”，支持单条/批量恢复、永久删除和清空。
- 恢复后回到原分类与标签；关联已不存在时给出可见的降级结果。
- 卡片/列表/详情、失效链接诊断和 MCP `delete_item` 统一走同一删除语义。
- MCP 删除成功响应明确返回 `movedToTrash: true`；未来可单独增加 restore 工具，但不是本版必须项。
- 本版不自动过期清理，避免新增后台计时和意外永久删除；由用户显式清空。

#### 最小数据与代码改动

- 给三类 item 增加可选 `deletedAt`，旧数据解码为 nil。
- 删除变成更新同一 `items.json` 中的条目状态，避免“活动 items.json → 独立 trash.json”跨文件移动的事务窗口。
- 普通搜索、分类计数、Agent 与 MCP 默认排除 `deletedAt != nil`；回收站查询只包含已删除条目。
- 只有永久删除成功后，才清理受控 `Images/` 内的 ImageItem 文件；Bookmark 的本地封面也应由同一清理规则覆盖，至少不能在移入回收站时删除。

#### 边界情况

- 恢复 Bookmark 时 URL 已被重新添加：阻止恢复并给出明确冲突，不覆盖现有条目；用户可永久删除回收站副本或先处理现有 Bookmark。
- 原分类已删除：恢复到 `None`，并显示“分类已重置”。
- 原标签已删除：丢弃无效标签引用，并显示丢弃数量。
- 外部/远程图片：永久删除只移除记录，不删除用户文件或远程资源。
- 重复删除：对已经在回收站的条目幂等，不更新原始删除时间。
- 批量与 MCP 删除：先验证所有目标，再提交批量更新；不能一半成功一半失败却统一返回成功。
- App 在写盘前崩溃：对外成功必须与可确认的持久化语义一致，不能沿用当前“排队写盘即成功”。

#### 非目标

- 不为 Category、Tag 提供回收站；它们的级联完整性应单独修正。
- 不提供任意编辑历史、版本对比或全局 Undo 栈。
- 不自动保留 30 天后清理；等持久化与通知策略稳定后再评估。

#### 验收门槛

- 所有现有 delete 入口都进入同一个回收站用例；没有旁路永久删除。
- 恢复后条目除 `deletedAt` 外的 metadata、notes、favorite、category、tags 和本地图片均保持一致。
- 永久删除不会越过内部 Images 目录，也不会删除仍被其他 Bookmark metadata 引用的封面。
- 批量删除/恢复遇到校验错误时保持原状态，并向 UI/MCP 返回可操作错误。
- 旧版 1.9.0 数据首次打开无需手工处理，失败时可回滚到原文件。

## 必须随版本完成的非功能工作

这些工作不应在发布说明中包装成“新功能”，但它们是上述功能可靠交付的条件：

1. **加入轻量 schema version 和显式迁移。** 新字段与智能集合不能继续只依赖一次性 Codable 解码；迁移失败要保留原文件并提供恢复路径。
2. **统一写入成功语义。** `DataStorage` mutation 应统一抛出/返回结果，调用者只在确认持久化后显示成功；无需为此更换 SQLite。
3. **收敛 Bookmark 富化流程。** 按 item ID 串行一次 OGP + 可选 AI，忙碌时 drain backlog，并至少显示 pending/failed/retry；消除两个快照互相覆盖。
4. **修复现有引用完整性。** 删除分类要处理 Bookmark/Image/Text，导入要重映射同名分类/标签 UUID。
5. **修复发布链。** CI 产物 glob 必须匹配 `dist/`，发布前运行 Swift/Helper 测试和 DMG/MCP smoke；公开 DMG 需要 Developer ID 签名与 notarization。
6. **解决 MCP Node 前置条件。** 正式分发若继续宣传 MCP，应提供自包含 runtime，或在设置页明确检测并报告 Node/路径问题。
7. **统一系统要求。** 工程实际最低 macOS 15.2，而 README/官网声称 13.0+；发布前必须选定并验证一个真实承诺。

## 后续版本建议

### 1.11 候选：系统 Share Extension

目标是从 Safari、Photos、Finder 和其他 App 的系统分享菜单直接保存 URL、图片或文本。它比专用浏览器扩展覆盖面更广，也符合 Seahorse 原生工具定位。

实施前先做一个技术 spike，验证 extension 与主 App 的 App Group/收件箱文件、去重、图片传递、主 App 未运行时的导入和失败重试。Extension 只负责落入可靠收件箱，类型识别与保存规则仍由主 App 复用；不要在 extension 中复制 `PasteHandler` 业务逻辑。

### schema 稳定后：图片 OCR 搜索

使用 Apple Vision 在本机提取截图/图片文字，并把 `extractedText` 纳入 `CollectionSearch`。先支持新图片异步 OCR 与手工“重建图片索引”，不上传图片、不引入额外 AI 服务。需要处理语言、旋转、大图、远程图片下载失败、索引版本和重建进度。

### 先验证再做：“重新发现”

可以先用一个简单原型展示随机的较旧收藏，验证用户是否会打开、保留或删除。若验证成立，再增加 `lastViewedAt`/`lastReviewedAt` 和更可靠的复习策略。当前没有查看历史，直接把“随机”包装成智能推荐证据不足。

### 单独大版本：真正跨设备同步

同步需要明确 source of truth、schema migration、图片传输、删除 tombstone、冲突解决、敏感设置边界、离线编辑和 iOS 发布流水线。当前“iOS 导入副本 + 移动 HTML”不应继续以同步措辞扩大承诺。先把本地模型、迁移和回收站稳定，再单独设计 CloudKit 或文件同步方案。

## 推荐交付顺序

1. 建立 schema version、迁移回滚和统一 mutation 结果。
2. 实现 `deletedAt`、全入口回收站与永久图片清理测试。
3. 扩展 `CollectionSearch.Criteria`，先用测试固定日期与 ANY/ALL 语义。
4. 实现 SmartCollection 持久化、Sidebar/UI 与导入导出。
5. 收敛 OGP/AI 富化竞态并补失败重试状态。
6. 完成真实数据迁移、1,000 条性能、批量/MCP、DMG 安装与发布门禁验证。

如果时间只能完成一个面向用户的主功能，应选择智能集合，但不能继续扩大 destructive 操作；回收站最迟必须与下一项批量/MCP 删除能力同时交付。

## 最终建议

Seahorse 不缺更多采集方式或 AI 演示，当前最缺的是让已有资料随着规模增长仍然可用、可控。1.10.0 应以智能集合为主卖点、回收站为安全底座，并把富化竞态、迁移和发布链作为不可见但必要的质量门槛。这样能用最少的新概念完成一个清晰的产品升级，同时为 Share Extension、OCR 和未来同步建立可信基础。

# 标准化 AI 书签解析流程

## 目标

把当前五次串行、自由文本驱动的 AI 解析改成一次结构化请求，并用本地确定性规则统一自动解析、批量解析、手动新增和主动重解析的分类与标签行为。

## 已确认的产品规则

- 分类是稳定的单选目录。AI 只能直接选择已有分类；没有合适项时保持未分类。
- 新分类建议只在手动新增或主动重解析时展示，并由用户确认后创建。自动与批量解析不创建分类，也不建立审核队列。
- 正常内容生成 2–4 个标签，优先复用已有标签；每次最多自动创建 2 个新标签。
- 新标签必须具体、可复用，不得与分类重复，也不得只是域名或站点名。
- 内容不足或判断不确定时允许返回空分类和不足 2 个标签，不强制补齐、不自动重试。
- 新标签使用用户设置的 AI 语言；已有标签保留数据库中的名称与大小写。
- 自动与批量解析只替换占位标题，并只填补空摘要、空分类和空标签，不覆盖人工内容。
- 用户主动重解析时展示标题、摘要、分类和标签的前后差异，由用户逐项确认。
- 核心解析协议与分类/标签规则由应用固定维护；设置只保留语言、自动创建新标签和附加偏好。

## 架构

保留现有网页抓取、OGP、富化状态和持久化流程。`AIManager` 用一次低温度、JSON Object 格式请求返回标题、摘要、分类、标签和 SF Symbol。网页内容作为不可信数据传入，system instruction 明确禁止执行网页中的指令。

模型输出先解码为 `ParsedBookmarkData`，再交给无副作用的 `BookmarkParsingPolicy`。策略层负责名称清理、分类白名单匹配、标签去重与过滤、已有标签优先、数量限制及新分类建议处理，输出 `BookmarkParsingResolution`。三条入口共享同一 resolution，只在最终交互方式上区分自动合并与用户确认。

不增加 provider protocol、向量检索、多阶段 Agent、分类审核队列、存储迁移或元数据来源字段。删除自动创建分类分支和 GitHub 分类特判。

## 输入与输出契约

输入包含 URL、原标题、清理后的正文、已有分类名称、已有标签名称、AI 语言和用户附加偏好。分类和标签列表以 JSON 数组传入，不使用逗号拼接。

模型只返回：

```json
{
  "title": "清理后的标题",
  "summary": "摘要",
  "categoryName": "已有分类名或 null",
  "newCategorySuggestion": "新分类建议或 null",
  "tags": ["tag-a", "tag-b"],
  "sfSymbol": "book.fill"
}
```

不使用模型自报的数值置信度。低置信度用 `null` 或空数组表达。请求、JSON 解码或持久化失败进入现有富化错误状态；单个建议为空不算失败。

## 本地解析规则

1. 清理分类名称后按大小写不敏感方式匹配已有分类；不匹配就保持未分类。
2. 新分类建议最多一个，只供交互流程展示。
3. 标签先清理首尾空白、合并连续空白并去重，再过滤与分类同名、与完整域名、父域或域名片段相同、属于泛化保留词、含换行或分隔符、长度异常的候选。
4. 已有标签优先于新标签；标签总数最多 4，新标签最多 2。
5. 匹配已有标签时使用数据库中的原始名称和 UUID。
6. SF Symbol 只有在系统能够解析时才使用，否则保留现有图标。

## 写回与交互

自动与批量解析使用同一缺失值合并规则：仅替换空标题或 `Loading...`、`Untitled` 等占位标题，并仅填补空摘要、未分类状态和空标签集合。新标签仍受“自动创建新标签”设置控制，但硬上限始终生效。

主动重解析完成后显示统一 diff sheet，只展示发生变化的字段。标题与摘要显示前后文本；分类显示旧值和建议值；标签显示保留、增加和移除。每行可独立选择，填补空值默认选中，覆盖已有值默认不选中。用户通过 `Apply Selected Changes` 提交。

设置页删除自动创建分类开关，把四个完整 prompt 编辑器收敛成一个附加偏好文本区。旧 UserDefaults prompt 值保留但不再读取，不做破坏性清理。

## 测试与验收

使用表驱动 XCTest 验证 JSON 解码、分类匹配、标签清理与上限、空结果、自动合并保护和主动 diff。测试只经过公开行为接缝，不 mock 内部类，也不调用真实模型；不对生成文字做脆弱快照。

完成标准：

- 每个书签只发起一次 AI 请求。
- 自动、批量和手动入口使用同一解析策略。
- 自动流程不覆盖人工元数据。
- 分类不会被自动创建。
- 标签总数与新标签上限由本地规则保证。
- 完整测试与 macOS Release 构建通过。

## 实施顺序

1. 以失败测试建立结构化解码与本地策略。
2. 改为单次 AI 请求并迁移自动、批量和手动新增入口。
3. 实现主动重解析 diff 与安全写回。
4. 收敛设置页，删除旧分支和特判。
5. 运行定向测试、完整测试、Release 构建与差异审查。
